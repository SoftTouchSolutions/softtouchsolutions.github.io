Generative Design Patterns -- Episode 1

Introduction

These series of posts looks at combining various design patterns in interesting ways. The unique combination of the design patterns comprise what can be dubbed as ‘Generative design patterns’.

The strategy pattern, which is placed into the behavioural category. The façade design pattern is a structural design pattern. The combination of these patterns is focus for today.

What is the strategy pattern?

The strategy pattern has been defined as encapsulating elements of an algorithm into a class. Or the encapsulation of a family of algorithms, such that they can be interchangeable and independent from clients that use them.

Strategy Pattern UML

The Strategy Pattern in UML looks like this:

Participating Structure

The Strategy patterns includes

  1. Strategy is the interface to be implemented by all ConcreteStrategy algorithms.
  2. ConcreteStrategy Various concrete algorithms supporting the general interface
  3. Context: This object is configured with a ConcreteStrategy object and maintains a reference to a Strategy object. It may also define an interface that lets Strategy access its data.

Tic-tac-toe Intent and Motivation

The strategy pattern, from the definition, helps to separate algorithms from clients that use them. This is a service-oriented architecture and separation of concerns. Within the SOLID framework the term Single Responsibility Principle (SRP).

Strategy, however, goes a step further by defining an interface for dependency injection for the family of algorithms separated from their calling classes and in likewise the calling clients may facilitate cohesion by defining a data access interface.

To apply the strategy pattern to our game, make the strategy to be Game. The algorithm interface is the game-loop because most if not all games will consist of this process. This has now succesfully abstracted a family of algorithms (game) and the concrete strategy (TicTacToe). This algorithm can now be called from a user interface client.

In the above image, the context class is the UIContext. This is because the GameStrategy is used in the user interface class of a flutter application.

Also note that in the above illustration. Only one strategy is implemented. In later patterns, we will be adding to the list of implementing game strategies. However, in we use the above concrete strategy as our default starting concrete strategy of the strategy pattern implementation.

abstract class IGameStrategy {
  IGameState gameState;
  IWinState winState

class TicTacToeGame implements IGameStrategy {
  @override
  List gameState = ['-','-','-','-','-','-','-','-','-'];
  @override
  List winStates = [[0,1,2],..]

  @override
  void gamePlay() {
    ...;
  }
}

The above code structure gives the general implementation for the strategy interface and the concrete strategy class. The generic strategy inteface being IGameStrategy and the concrete strategy here is the TicTacToeGame.
