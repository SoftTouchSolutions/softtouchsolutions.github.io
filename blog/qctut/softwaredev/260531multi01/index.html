Multi-Processing Series - Episode 1

Introduction

This series brings a fresh take on multiprocessing architecture. Gordon Moore’s 1965 paper “Cramming More Components onto Integrated Circuits” [17] observed that the number of transistors that could be placed on an integrated circuit was doubling approximately every two years. For several decades, that transistor density increase translated almost automatically into faster clock speeds and faster single-threaded execution. A new chip generation meant faster software for free. Herb Sutter’s 2005 article “The Free Lunch Is Over” [18] was the first widely-read industry account of what was happening at the hardware frontier: the clock-speed wall had arrived. As frequencies approached and exceeded 3 GHz, heat dissipation became the binding constraint. The faster the clock, the more power each transistor switching event consumes, and at the switching rates required to push beyond roughly 3 GHz the silicon simply becomes too hot to operate reliably. Transistor counts have continued to increase since then, but the gains are now channelled into additional cores, wider vector units, and larger caches rather than faster clocks. The practical ceiling for sustained CPU operation sits at around 2.5–3 GHz for the vast majority of deployed hardware, with brief boost frequencies topping out around 5–6 GHz only in short thermal headroom on high-end desktop silicon.

The thrust of the story is that since Moore’s Law has plateaued in the dimension of clock speed, the way computers have kept pace with performance demand is largely through multiprocessing: running more things at once, on more cores, simultaneously. That is the subject of this series of tutorials. In this first episode we go over multiprocessor hardware architecture, memory models, and the IRIW (Independent Reads of Independent Writes) litmus test, which is a classical example that distinguishes sequentially consistent memory models from weaker ones.

This series is distinguished from most multiprocessing tutorials by analysing the topic through the lens of specific programming environments and language paradigms. Rather than treating concurrency as an abstract topic, we look at the same hardware realities through eight different programming lenses: standard C++, Qt C++, C, FreeRTOS/C, Java, C#, Go, Rust, and F#. Each of these environments exposes the underlying hardware behaviour differently. Some abstract it away entirely; others require the programmer to manage it directly. By examining the same phenomena across all eight, we build a well-rounded and honest picture of what is invariably a low-level system programming topic. The companion book to this series covers Chapters 1 through 7 at full depth; this episode focuses specifically on the hardware foundations that Chapter 2 of the book introduced — and extends it with the Qt C++ and FreeRTOS/C treatment that that chapter did not cover.

The Hardware That Makes Concurrency Hard

Modern CPUs execute instructions out of order, buffer writes in store queues, and maintain per-core caches. These optimizations are transparent within a single thread but produce counter-intuitive behaviour when threads share mutable state. Understanding why requires stepping inside the processor.

Beginning with superscalar designs in the 1990s, CPUs have exploited instruction-level parallelism: they analyse a window of upcoming instructions, identify those whose inputs are already available, and dispatch them to independent execution units simultaneously — integer arithmetic, floating-point, memory loads, and memory stores can all proceed at once. An out-of-order processor does not wait for earlier instructions to complete before issuing later ones that are independent. The critical safety mechanism is retirement in order: even though instructions may execute out of sequence, the processor always commits their results to the architectural state (registers, memory) in the original program order. A single thread therefore sees behaviour that is indistinguishable from purely sequential execution. The problem arises only when a second thread on a second core enters the picture.

One of the most consequential optimizations for concurrent programmers is the store buffer. When a CPU core executes a store instruction, the write does not propagate immediately to the cache. Instead, it lands in a small per-core buffer of pending writes, and the core continues executing without waiting for the write to drain. The store buffer empties asynchronously. A subsequent load from the same address within the same thread is satisfied directly from the store buffer, so the thread always sees its own most recent writes. But a thread on a different core reads the cache, which may still hold the old value because the first core’s store buffer has not yet flushed. McKenney [1] provides the definitive hardware-level account of store buffers, with detailed barrier tables for x86, ARM, and POWER that map each reordering type to the barrier instruction needed to prevent it.

The hardware is not the only source of reordering. Compilers perform aggressive optimizations — hoisting reads out of loops, eliminating redundant loads, reusing register values across what appear to be independent reads — all of which are entirely valid under single-threaded semantics. The moment shared mutable state is involved, these transformations can silently break correctness. The classic example is a spin-wait in C# where the ready flag is never written inside the loop body:

// C# — naive spin wait; optimizer may hoist the read, producing an infinite loop
bool ready = false;
// Thread 1
while (!ready) { }

// Thread 2
ready = true;

A sufficiently aggressive JIT compiler can observe that ready is never written within the loop body on Thread 1, conclude that its value cannot change during the loop, and hoist the read entirely out of the loop, producing an infinite loop regardless of what Thread 2 does. This is correct single-threaded behaviour. Preventing it requires marking ready as volatile, which tells the compiler that this variable may be modified by another thread and its value must be reloaded on every access. Sutter [3] examines exactly this class of compiler transformation and explains why the C++11 memory model was designed to prevent it.

In C, the same hazard appears in FreeRTOS applications where a task polls a flag set by an ISR or by another task. The volatile keyword in C is both necessary and sufficient to prevent the compiler from eliminating or caching the read, but it does not insert hardware memory barriers. That distinction matters enormously on ARM, as we will see.

Cache Coherence, the MESI Protocol, and False Sharing

Multiple cores that each maintain private caches must agree on the value of every shared memory location. The hardware mechanism for this agreement is the cache coherence protocol. The most widely deployed variant is MESI, named for the four states a cache line can occupy: Modified (written by this core, not yet flushed — only valid copy in the system), Exclusive (clean, present in only this cache, consistent with main memory), Shared (clean, present in multiple caches), and Invalid (stale — a write by another core has invalidated this copy). When one core writes to a cache line in the Shared state, the coherence protocol broadcasts an invalidation message to all other caches holding that line, transitioning them to Invalid. The writing core’s line transitions to Modified. Any other core that subsequently reads the same address incurs a cache miss and must fetch the updated copy, either from the writing core’s cache directly or from main memory. McKenney [1], Appendix C, provides the full MESI state machine specification including the MOESI and MESIF variants found in AMD and Intel systems respectively.

The cache coherence protocol operates at cache line granularity, typically 64 bytes on modern x86 and ARM processors. This creates an insidious performance hazard known as false sharing: two threads on different cores can write to entirely different variables that happen to reside on the same 64-byte cache line. The threads share no logical data, but the coherence protocol treats every write to that line as a conflict, broadcasting invalidation messages and causing repeated cache misses. The threads are fighting over a piece of silicon rather than any meaningful shared state. McKenney [1] demonstrates false sharing empirically, with benchmark results showing throughput degradation of five to ten times on representative hardware.

In Qt C++, the remedy is to ensure that independently written variables occupy separate cache lines. The alignas specifier from C++11 guarantees placement at a cache line boundary:

// Qt C++ — eliminating false sharing between two worker counters
struct alignas(64) AlignedCounter {
    QAtomicInt value{0};
};

AlignedCounter counter0;   // lives on its own cache line
AlignedCounter counter1;   // lives on a different cache line

// Thread 0 writes counter0, Thread 1 writes counter1 — no false sharing

In FreeRTOS/C, the same problem appears when multiple tasks share a struct whose fields are each owned by a different core. The GCC and Clang compilers used for ARM targets support __attribute__((aligned(N))):

/* FreeRTOS C — two counters, each pinned to its own cache line */
typedef struct {
    volatile int value __attribute__((aligned(64)));
} AlignedCounter_t;

static AlignedCounter_t g_counter0;   /* Core 0 writes this */
static AlignedCounter_t g_counter1;   /* Core 1 writes this */

Without the alignment attribute, a naive struct { int c0; int c1; } places both counters in the same 64-byte line, and every increment by either core invalidates the other core’s cached copy — coherence traffic for nothing.

Memory Barriers: Telling the Hardware What You Mean

Given that both hardware and compilers reorder memory operations, concurrent programs require explicit mechanisms to enforce ordering when correctness depends on a specific visibility sequence. These mechanisms are called memory barriers or memory fences. Sutter [3] is the clearest available explanation of why barriers are necessary and how each C++ memory ordering mode maps to machine instructions on x86, ARM, and POWER.

A release store is a store preceded by a release fence: all stores that appear before it in program order are guaranteed to become visible to any thread that subsequently observes the released value. An acquire load is a load followed by an acquire fence: once a thread observes the value written by a paired release store, all subsequent loads in that thread are guaranteed to see all writes that were ordered before the release. Together, the release store and acquire load establish a happens-before edge across threads. This is the minimum synchronization needed for a producer-consumer relationship, and it is the pattern underlying every mutex, semaphore, and queue in every environment this series covers.

In Qt C++, the QAtomicInt API exposes this distinction directly through its naming convention. The storeRelease and loadAcquire methods correspond exactly to C++11’s memory_order_release and memory_order_acquire:

// Qt C++ — acquire-release producer-consumer
static QAtomicInt g_ready{0};
static QString    g_data;

// Producer thread
g_data = QStringLiteral("sensor reading");
g_ready.storeRelease(1);      // release: g_data write visible before this

// Consumer thread
while (g_ready.loadAcquire() == 0) {
    QThread::yieldCurrentThread();
}
// g_data is guaranteed visible here — the acquire fence orders reads after it
qDebug() << g_data;

In FreeRTOS/C on an ARM Cortex-A target — the architecture that matters for multi-core embedded systems — the equivalent requires explicit ARM barrier instructions. The ARM DMB SY (Data Memory Barrier, full system) is the release/acquire-equivalent: it ensures that all explicit memory accesses before the barrier complete before any explicit memory accesses after it are observed by any other processor in the system [21]:

/* FreeRTOS C — acquire-release on ARM Cortex-A using explicit DMB */
static volatile int g_data  = 0;
static volatile int g_ready = 0;

/* Producer task */
static void vProducerTask(void *pvParameters) {
    for (;;) {
        g_data = readSensor();
        __asm volatile("dmb sy" ::: "memory");  /* release fence */
        g_ready = 1;
        vTaskDelay(pdMS_TO_TICKS(10));
        g_ready = 0;
    }
}

/* Consumer task */
static void vConsumerTask(void *pvParameters) {
    for (;;) {
        while (g_ready == 0) { taskYIELD(); }
        __asm volatile("dmb sy" ::: "memory");  /* acquire fence */
        processData(g_data);                    /* g_data visible here */
    }
}

Two things are worth noting about the FreeRTOS example. First, volatile in C prevents the compiler from caching or eliminating the accesses to g_ready and g_data, but it does not emit any hardware barrier instruction. On a single-core Cortex-M3 or M4, volatile alone is sufficient because there is only one core and all memory operations from its perspective are already globally ordered. On a multi-core Cortex-A target, the hardware can still reorder stores to independent addresses between cores; only an explicit DMB SY closes that gap. Second, in real FreeRTOS applications the correct approach is to use FreeRTOS queues, mutexes, or event groups for inter-task communication — these primitives include the necessary hardware barriers in the FreeRTOS kernel implementation itself, so application code does not need to manage them manually. The raw barrier examples here exist to make the hardware behaviour concrete, not to serve as a production pattern.

Memory Models Across Languages — and the Qt/FreeRTOS Gap

Every language studied in this series defines a memory model: a formal contract between the programmer, the compiler, and the hardware that specifies which reorderings are permitted and which synchronization operations establish happens-before edges. The models differ substantially in what they expose and what they hide.

C++ provides the most explicit and flexible model [5], with six distinct memory ordering modes ranging from memory_order_relaxed (atomicity only, no ordering constraints) through memory_order_acquire/memory_order_release (one-sided barriers) to memory_order_seq_cst (full sequential consistency). Rust [7] mirrors the C++ model exactly, adding compile-time prevention of data races via the Send and Sync marker traits — a data race in Rust safe code is a compile error, not a runtime failure. Java’s model [9, 10], specified formally by Manson, Pugh, and Adve [11], defines visibility in terms of happens-before edges; volatile writes are release operations, volatile reads are acquire operations, and the VarHandle API introduced in Java 9 exposes the full ordering spectrum explicitly. Go [13] takes the simplest approach: all sync/atomic operations are sequentially consistent, channel operations carry implicit acquire-release semantics, and there is no weaker atomic ordering available. C# and F# [12, 4] sit on the CLR, where volatile fields carry acquire-release semantics and Interlocked operations are fully sequentially consistent. F# additionally offers the MailboxProcessor agent pattern [2, 14], which routes all mutable state through a single-threaded message loop and sidesteps memory ordering concerns entirely for that state.

Qt C++ and FreeRTOS/C occupy a different and instructive position, which is the main gap this episode fills relative to the companion book’s Chapter 2 treatment.

Qt C++ builds on C++ and the host operating system’s threading layer (pthreads on Linux and macOS, Win32 threads on Windows), but it provides its own atomic types — QAtomicInt, QAtomicPointer<T>, and QBasicAtomicInt — whose API deliberately names the acquire-release distinction. The Qt documentation [19] defines three tiers: loadRelaxed/storeRelaxed provide atomicity with no ordering constraints; loadAcquire/storeRelease provide one-sided barriers; and the fetchAndAddOrdered family and std::atomic with default ordering provide full sequential consistency. A Qt programmer who reaches for loadAcquire/storeRelease is using acquire-release semantics and is subject to the same IRIW-style outcomes on ARM hardware as a C++ programmer using explicit memory_order_acquire/memory_order_release. This is not a flaw — acquire-release is correct for producer-consumer relationships — but it is not enough for every synchronization pattern, and understanding where it falls short is exactly what the IRIW test reveals.

FreeRTOS/C operates below any language memory model. It is a C-based RTOS that provides no atomic types at the language level. The memory ordering guarantees in a FreeRTOS application depend entirely on three things: the FreeRTOS primitives themselves (queues, mutexes, event groups, and semaphores all include the necessary barriers in the kernel implementation), the volatile keyword for compiler ordering, and explicit ARM barrier instructions for hardware ordering on multi-core Cortex-A targets [21]. The separation between compiler ordering and hardware ordering — two independent concerns that most language memory models merge into a single abstraction — is fully exposed in FreeRTOS C and demands that the programmer understand both.

The IRIW Litmus Test

The Independent Reads of Independent Writes (IRIW) litmus test is the classical example that distinguishes sequentially consistent memory models from weaker ones. It involves four threads and two shared variables, x and y, both initially zero:

  • Thread 1 writes x = 1.
  • Thread 2 writes y = 1.
  • Thread 3 reads x (obtaining r1) then reads y (obtaining r2).
  • Thread 4 reads y (obtaining r3) then reads x (obtaining r4).

The question is: can Thread 3 observe r1=1, r2=0 (x visible, but y not yet) while Thread 4 simultaneously observes r3=1, r4=0 (y visible, but x not yet)? In other words, can two observer threads disagree on the order in which the two independent writes occurred? Adve and Gharachorloo [15] use IRIW as a canonical example for distinguishing sequential consistency from weaker consistency models. McKenney [1], Chapter 15, shows the specific barrier instructions required to prevent or permit this outcome on x86, ARM, and POWER.

Under sequential consistency, as formalized by Lamport [8], the answer is no. All threads must agree on a single total order of writes. If Thread 3 sees the write to x before the write to y, then that total order mandates that every thread sees the write to x before the write to y. Thread 4 cannot simultaneously observe y=1 while x=0. This is the intuitive mental model most programmers apply when they reason about multithreaded code — and on many hardware platforms it simply does not hold.

On POWER and ARM processors, IRIW outcomes are observable in practice, as documented by McKenney [1] with concrete litmus test programs and assembly output. On x86, the Total Store Order (TSO) model ensures that all processors observe stores in the same global order, though each processor may observe its own stores early; IRIW outcomes are therefore not possible on x86 without software assistance. On ARM’s weaker model, a store can become visible to some cores before others, making IRIW outcomes possible without explicit barriers.

The IRIW test directly motivates the sequentially consistent ordering mode in every language memory model this series covers. When you use memory_order_seq_cst in C++, SeqCst in Rust, or sync/atomic in Go [13], the compiler emits the barriers necessary to prevent IRIW outcomes on any target architecture, at the cost of additional synchronization overhead. When you use acquire-release, no such guarantee is made: IRIW-style disagreements are permitted, and programs that require all observers to agree on a single write order must use sequential consistency. Sutter [3] demonstrates this with assembly output for both x86 and ARM, showing that sequential consistency requires an MFENCE on x86 and a full DMB SY on ARM that the acquire-release mode can omit.

For C# and F# programmers working with Volatile operations: CLR volatile reads and writes provide acquire-release semantics, not full sequential consistency, as specified in ECMA-335 [12]. Programs that require all threads to agree on a single total order of writes across multiple variables must use Interlocked operations instead, which the .NET runtime documentation [4] specifies as sequentially consistent full fences.

IRIW in Qt C++: Acquire-Release vs. Sequential Consistency

Qt C++‘s QAtomicInt makes it possible to construct the IRIW scenario precisely and then show how the choice of ordering mode changes the outcome. The following example uses four QThread-based workers:

// Qt C++ — IRIW with acquire-release (outcome is possible on ARM)
#include <QAtomicInt>
#include <QRunnable>
#include <QThreadPool>
#include <QDebug>
#include <atomic>

static QAtomicInt g_x{0}, g_y{0};
static int g_r1{-1}, g_r2{-1};   // Thread 3 observations
static int g_r3{-1}, g_r4{-1};   // Thread 4 observations

struct WriterX : QRunnable {
    void run() override { g_x.storeRelease(1); }
};

struct WriterY : QRunnable {
    void run() override { g_y.storeRelease(1); }
};

// Observer A: read x then y
struct ReaderXY : QRunnable {
    void run() override {
        g_r1 = g_x.loadAcquire();
        g_r2 = g_y.loadAcquire();
        // On ARM: g_r1=1, g_r2=0 is possible — x visible, y not yet
    }
};

// Observer B: read y then x
struct ReaderYX : QRunnable {
    void run() override {
        g_r3 = g_y.loadAcquire();
        g_r4 = g_x.loadAcquire();
        // On ARM: g_r3=1, g_r4=0 simultaneously — the IRIW outcome
    }
};

With storeRelease/loadAcquire, each observer establishes a happens-before edge only with the specific writer whose released value it read. There is no constraint that forces the two observers to see the two independent writes in the same order. On an ARM target — where Qt applications are commonly deployed: Android phones, embedded Linux boards (Raspberry Pi, STM32MP1, NXP i.MX), and ARM-based server chips — Thread 3 may observe x=1 while y=0, and Thread 4 may simultaneously observe y=1 while x=0. The two observers genuinely disagree on the write order.

To prevent this, all four operations must use sequential consistency. In Qt C++, the cleanest approach is to use C++11 std::atomic<int> with its default ordering, which is memory_order_seq_cst:

// Qt C++ — IRIW prevented with sequential consistency
#include <atomic>
#include <QRunnable>

static std::atomic<int> g_x{0}, g_y{0};

struct WriterX : QRunnable {
    void run() override { g_x.store(1); }   // seq_cst by default
};

struct WriterY : QRunnable {
    void run() override { g_y.store(1); }
};

struct ReaderXY : QRunnable {
    int r1{-1}, r2{-1};
    void run() override {
        r1 = g_x.load();   // seq_cst
        r2 = g_y.load();   // seq_cst — IRIW outcome now impossible
    }
};

struct ReaderYX : QRunnable {
    int r3{-1}, r4{-1};
    void run() override {
        r3 = g_y.load();
        r4 = g_x.load();
    }
};

With sequential consistency, all four threads participate in a single global total order of writes. The IRIW outcome is structurally prevented: if ReaderXY observes x=1, the total order mandates that x’s write precedes y’s write for all threads, so ReaderYX cannot simultaneously observe y=1 while x=0.

The performance trade-off is real. On ARMv8, a sequentially consistent store compiles to STLR (store-release with total-order constraint) or to a STR followed by DMB ISH depending on the microarchitecture. The acquire-release variant compiles to STLR without the reciprocal load-ordering constraint. On x86, both compile to MOV with no additional fence because TSO handles the ordering for free. The practical advice for Qt developers is the same as for C++: use loadAcquire/storeRelease (or memory_order_acquire/memory_order_release) when the only requirement is that one thread’s writes are visible to another after a specific synchronization point. Use sequential consistency when the correctness of the program requires all threads to agree on the order of multiple independent writes.

IRIW in FreeRTOS/C on ARM: When the Hardware Can Disagree

FreeRTOS applications running on single-core ARM Cortex-M targets — the M0, M3, M4, and M7 processors found in the vast majority of STM32, Nordic, and NXP microcontrollers — are not subject to IRIW in hardware. A single core is the only observer; there is no second core to disagree with. On these targets, the IRIW concern is purely a compiler-reordering concern, and volatile combined with standard FreeRTOS synchronization primitives is sufficient.

The IRIW hazard becomes real on multi-core ARM Cortex-A targets, which is where FreeRTOS SMP — the symmetric multiprocessing variant introduced in FreeRTOS V10.4.0 — operates. Cortex-A9 and Cortex-A53 processors implement a weakly ordered memory model [21] and are found in widely used platforms: the STM32MP157 (A7 dual-core), NXP i.MX6 (A9 dual-core to quad-core), Raspberry Pi 3 and 4 (A53 and A72). On these targets, independent stores from different cores can become visible to observer cores in different orders unless explicit DMB SY barriers are inserted.

The following example replicates the four-task IRIW scenario in FreeRTOS C, first in the vulnerable form and then in the corrected form:

/* FreeRTOS C — IRIW on ARM Cortex-A SMP: the vulnerable version */
#include "FreeRTOS.h"
#include "task.h"

static volatile int g_x = 0, g_y = 0;
static volatile int g_r1 = -1, g_r2 = -1;   /* Task 3 observations */
static volatile int g_r3 = -1, g_r4 = -1;   /* Task 4 observations */

/* Task 1 (Core 0): write x with a release fence */
static void vWriterX(void *pvParameters) {
    g_x = 1;
    __asm volatile("dmb sy" ::: "memory");   /* release: g_x=1 ordered before flag */
    vTaskDelete(NULL);
}

/* Task 2 (Core 1): write y with a release fence */
static void vWriterY(void *pvParameters) {
    g_y = 1;
    __asm volatile("dmb sy" ::: "memory");   /* release: g_y=1 ordered before flag */
    vTaskDelete(NULL);
}

/* Task 3 (Core 0): acquire-release reads only — IRIW vulnerable */
static void vReaderXY(void *pvParameters) {
    __asm volatile("dmb sy" ::: "memory");   /* acquire before first read */
    g_r1 = g_x;
    /* No barrier between the two reads — IRIW outcome possible here */
    g_r2 = g_y;
    /* g_r1=1, g_r2=0 is observable on ARM Cortex-A */
    vTaskDelete(NULL);
}

/* Task 4 (Core 1): acquire-release reads only — IRIW vulnerable */
static void vReaderYX(void *pvParameters) {
    __asm volatile("dmb sy" ::: "memory");
    g_r3 = g_y;
    g_r4 = g_x;
    /* g_r3=1, g_r4=0 simultaneously with Task 3's g_r1=1, g_r2=0 — the IRIW outcome */
    vTaskDelete(NULL);
}

The troubling outcome — g_r1=1, g_r2=0 on Core 0 concurrent with g_r3=1, g_r4=0 on Core 1 — is permitted by the ARM memory model [21] when each observer has only an acquire fence before its first read. Each observer establishes a happens-before edge with the single writer whose write it observed, but there is no global ordering constraint that forces both observers to agree on the relative order of two independent writers. The cores see the world from different angles, and without an explicit full-system barrier to synchronize those views, they can tell different stories.

To restore sequential consistency, the corrected version places a full DMB SY between the two reads in each observer task:

/* FreeRTOS C — IRIW prevented: full-system barrier between the two reads */

/* Task 3: corrected */
static void vReaderXY_Fixed(void *pvParameters) {
    g_r1 = g_x;
    __asm volatile("dmb sy" ::: "memory");   /* full barrier between the two reads */
    g_r2 = g_y;
    /* IRIW outcome is now impossible */
    vTaskDelete(NULL);
}

/* Task 4: corrected */
static void vReaderYX_Fixed(void *pvParameters) {
    g_r3 = g_y;
    __asm volatile("dmb sy" ::: "memory");
    g_r4 = g_x;
    vTaskDelete(NULL);
}

The distinction between the two versions is subtle but critical. In the vulnerable version, the DMB SY before the first read orders that read with respect to the writer it observes — a two-thread relationship. In the corrected version, the DMB SY between the two reads forces the hardware to drain all pending memory operations across the full system before proceeding to the second read, which is the ARM equivalent of what memory_order_seq_cst compiles to. Both observers are now looking at the same globally ordered snapshot of memory, and the IRIW outcome becomes structurally impossible.

For production FreeRTOS code, the recommendation remains to use FreeRTOS queues, mutexes, or event groups for all inter-task communication. The raw barrier examples above illustrate the hardware behaviour; they are not intended as a template. FreeRTOS’s own synchronization primitives already include the correct barriers and spare the application developer from reasoning about them directly.

Data Races, DRF-SC, and Why the Primitives Matter

Data races are the failure mode of incorrect synchronization. Two actions constitute a data race if they access the same memory location, at least one of them is a write, and neither happens-before the other. In C++, a data race is undefined behaviour: the compiler is legally entitled to assume data races do not occur, and the optimizations it makes on that assumption can produce behaviour completely unrelated to any sequential reading of the code [5]. A data race in C++ is not merely incorrect; it can corrupt memory, produce wrong computations, or create security vulnerabilities. In the JVM and CLR, a data race produces indeterminate but not arbitrary values — a Java or C# program with a data race will not produce a completely random bit pattern, but the value a thread reads may correspond to a value written by any thread at any prior point [10, 12]. In Rust (safe code), the type system prevents data races at compile time [7]. In Go, the built-in race detector (go run -race) diagnoses races at runtime [13].

The DRF-SC guarantee — Data Race Free programs execute with Sequential Consistency — was first formalized by Adve and Hill [16] and is the organizing principle of the C++11, Java, and CLR memory models. Adve and Gharachorloo [15] explain its significance clearly: if a program correctly establishes happens-before edges between all conflicting accesses through the synchronization primitives of its language, the program executes sequentially consistently, even on hardware that permits extensive reordering. The complex machinery of memory orders, fences, and acquire-release pairs exists precisely to make programs data-race-free. Get that right, and the hardware complexity becomes invisible.

For FreeRTOS developers, this is an important framing. The FreeRTOS queue, mutex, semaphore, and event group implementations all include the platform-correct memory barriers in the kernel itself. A FreeRTOS application that communicates exclusively through these primitives is data-race-free by construction on any supported architecture, including weakly ordered ARM Cortex-A. The IRIW hazards described in the previous section only arise when application code bypasses those primitives and uses raw shared variables with manually placed barriers — a technique that is occasionally necessary for performance-critical ISR-to-task signalling, but requires careful attention to memory ordering.

What IRIW Means in Practice

For the majority of Qt C++ developers targeting desktop, mobile, or embedded Linux on x86-64 or ARMv8, the IRIW concern is real but contained. On x86-64, TSO prevents IRIW at the hardware level. On ARMv8 (Android, Apple Silicon, ARM server and embedded boards), IRIW is a hardware possibility, and the choice between loadAcquire/storeRelease and std::atomic default sequential consistency has measurable performance consequences on highly contended paths — Sutter [3] quantifies this for both architectures. The practical rule is: use acquire-release when the requirement is that one thread’s writes become visible to another after a specific synchronization point; use sequential consistency when the program requires all threads to agree on the relative order of multiple independent writers.

For FreeRTOS developers on multi-core Cortex-A targets, the IRIW concern is more immediate because C gives you no memory model at all — only volatile for compiler ordering and raw hardware barriers for hardware ordering. The safest path is to use FreeRTOS primitives for all inter-task communication and never share raw global variables without either a mutex or an explicit release-acquire barrier pair at both the production and consumption sites.

Understanding IRIW is not an academic exercise. It is the precise diagnostic that tells you whether the memory model guarantee you are relying on is strong enough for your synchronization pattern. It maps directly onto the hardware and API choices available in every environment this series examines, and in the next episode we will see how the same tension — between acquire-release efficiency and sequential consistency correctness — drives the design of every synchronization primitive from mutexes to message queues to atomic counters.
